#caffe 源码阅读之data_transformer

今天来说说data_transformer,data_transformer是caffe的基础类.
正如其名,它要做的事情就是对输入数据进行转换,例如对输入图片的进行crop mirror,减中值啊之类.
也可以根据需要开发新的转换方向.

1 主要成员变量和方法

首先是构造函数,需要指定Transformation 和 Phase,后者是定义训练类别,比如train test
前者是caffe.proto里面定义的数据结构,定义了scale mean_file mean_value crop等

InferBlobShape 这一系列方法是通过传入的数据以及TransformationParameter来推测输出数据的BlobShape
暂且不表.

Transform 这一系列方法是将不同的数据格式转换成Blob形式.
拿最简单的Datum -> Dtype* 来说,其他的都大同小异.

首先 mean_value 在caffe.proto中属于repeated类型,可以指定一个或者跟channels数严格相等个,并且不能和
mean_file同时指定.

这组数据是否做镜像,依据的是TransformationParameter 的mirror参数设为true,并且同时随机数随机成1

首先如果设定了mean_values的话，要么设置一个value值，然后复制datum_channels份，所有的通道共同使用这个，要么设定datum_channels个．

其中h_off, w_off分别是进行crop时候，crop后图像数据的原点(也就是左上角)在原图像中的h坐标和w坐标．

值得注意的是，通过mirror和crop，caffe可以轻易地增加大量training data.

之后就是进行crop后的数据和原图像数据进行减中值和scale处理．

近来在使用caffe实现gru unit时候需要书写梯度反向传播的部分代码，而总是弄混caffe中关于gemm的部分代码
因此需要将blas彻底弄清楚，才不会浪费更多的时间，是以记之。

blas以及他的各种衍生包的安装舍去不表。

blas 是 Basic Linear Algebra Subprograms的简写，主要进行的是矩阵运算（向量也算特殊的矩阵）。
因此按照维度的增加，blas将运算分为三个level

向量与向量的运算
矩阵和向量的运算
矩阵和矩阵的运算

（由于太多，现在只了解用到部分，其余部分随用随添加）

  1. level1 向量向量运算
    cblas_?asum ？可取s, d, sc, dz 作用是Sum of vector magnitudes （参数）
  2. level2 矩阵向量运算

  3. level3 矩阵矩阵运算
    cblas?gemm ？ 可取s, d, c, z 用作一般矩阵的乘法
    void cblas    ？gemm (const CBLAS_LAYOUT Layout, const CBLAS_TRANSPOSE transa, const CBLAS_TRANSPOSE transb, 

    const MKL_INT m, const MKL_INT n, const MKL_INT k, const float alpha, const float *a, 
const MKL_INT lda, const float *b, const MKL_INT ldb, const float beta, float *c, const MKL_INT ldc);

    C := alphaop(A)op(B) + beta*C,
    alpha beta都是scalar A,B,C是矩阵 A（m×k） B（kxn） C（mxn）
    如果transa=CblasNoTrans, op(A) = A， transa=CblasTrans, then op(A) = AT，transa=CblasConjTrans, then op(A) = AH.
    m 指定的是op（A）以及C的行， n指定的是op(B)和C的列， k是op（A）的列和op（B）
    lda 之类指定的是相应向量的leading demension 比如rowmajor下，transa=CblasNoTrans，lda>=max(1, k)

首先简易说下caffe.cpp的使用方法

./build/tools/caffe train -solver examples/test/lenet_solver.prototxt -gpu 0
这是最简单的训练方法，其中-gpu 0 设定了device number, 如果不使用这个参数的话，程序会检查lenet_solver中是否指定mode 为 GPU，已经device num
./build/tools/caffe train -solver examples/test/lenet_solver.prototxt -gpu 0 -snapshot examples/test/lenet_solver_iter_75.solverstate
这是接着上次的训练，比如因为其他内在外在的原因，其中snapshot的参数为保存的以前的训练状态，例如上式中iter_75就是训练到75的时候
./build/tools/caffe train -solver examples/test/lenet_solver.prototxt -gpu 0 -weights examples/test/lenet_solver_iter_75.caffemodel
这是finetune的过程，其中.caffemodel是之前训练保存的参数。finetune真是好用，简单不到10次重复finetune就达到了我之前使用几万次迭代才能得到的结果

./build/tools/caffe device_query -gpu 0
查询制定的gpu的参数。

./build/tools/caffe test -model examples/test/lenet_train_test.prototxt -weights examples/test/lenet_solver_iter_75.caffemodel -gpu 0 -iterations 100
这是测试模型-model参数指的是神经网络模型的net参数 -weights 为训练出来的网络参数 -iterations为测试的迭代次数 默认为50

今天来说说caffe中的底层类SyncedMemory

首先SyncedMemory是caffe的内存管理类，由他实现了cpu data和gpu data的同步。

  1. SyncedMemory的成员变量

    void* cpu_ptr_; void* gpu_ptr_; size_t size_; SyncedHead head_; //enum SyncedHead { UNINITIALIZED, HEAD_AT_CPU, HEAD_AT_GPU, SYNCED }; bool own_cpu_data_; bool cpu_malloc_use_cuda_; bool own_gpu_data_; int gpu_device_;
    从名字就可以一目了然知道每个成员变量的具体含义，尤其需要注意的是own_[c,g]pudata,这两个变量标示的是
    如果相对应的[c,g]puptr指向的数据是通过malloc等方式申请的话，其值为true，通过set_方法赋值时（与别的变量共享），其值为false
    从而在变量销毁的时候不会将共享数据的数据释放掉。
    cpu_malloc_usecuda 变量标示的是在申请cpu内存的时候是否使用的是cuda的相应方法，使用cuda的方法能够申请得到pinned memory,是的
    在多卡进行训练时候，提高数据copy的效率使得large models on multi-gpu 更加稳定。

  2. SyncedMemory的成员函数
    private 成员函数,
    void to_cpu(); void to_gpu();
    这俩函数是类内部函数，实现的是cpu和gpu数据的同步。值的注意的地方是在 在数据传输过程中，数据head_的更新。

    接口函数
    `
    const void cpu_data();
    void     mutable_cpu_data();
    void set_cpu_data(void* data);

    const void gpu_data();
    void     mutable_gpu_data();
    void set_gpu_data(void* data);

    size_t size();
    SyncedHead head();
    void async_gpu_push(const cudaStream_t& stream);
    `
    很简易的接口，从名字就可以知道他们的作用。

  2. size相关的

    size() 和　length()都返回的是字符串的大小，但更推荐用前者。capacity() 返回的是为当前字符串申请的空间的大小，这个
    大小是以byte来记，并且这个大小不小于字符串的长度。max_size()返回的是字符串能申请到的最大长度。因为字符串可以动态改变大小
    所以，为了更加有效率，当可以预先知道字符串的大致长度范围的时候，最好使用reserve函数申请合适的空间长度，防止字符的不停
    拷贝。
    empty() 返回的是字符串是否为空，当然你可以使用size() <= 0来判断，但是还是推荐使用这个函数。

使用gdb进行调试

#缘起
要在ubuntu下工作，又不太想把程序都放在window上写再来回转着调试，所以就想联系在linux系统里使用gdb进行调试

#相信使用

  1. 编译时加入debug信息, 使用参数-g即可。
    g++ -g main.cpp -0 main

  2. 启动测试
    可以使用gdb main，使用-tui 参数可以分屏显示代码，比参数l好用的要死
    也可以使用 gdb 打开调试窗口后，使用file app来打开app进行调试

    set args 来设定程序运行的参数
    也可以gdb之后，使用run <参数列表>
    show args显示参数

  3. 各种参数
    l list 列出当前行前后个5行代码 list n1 n2 列出n1 n2 之间的代码
    c continue 继续运行
    r run
    b break 设置断点 后面可以接行号 以及函数名字；也可以条件断点break 46 if testsize==100；break file：N 设置file的N行为断点
    p 打印后面变量的名字， print &var 打印变量的地址；如果a是一个数组，10个元素，如果要显示则： print *a@10
    n next 单步运行一次， next N，执行下N步

    s 跳入函数内部
    finish 执行完当前运行的函数，返回调用他的函数
    j jump 5 跳转到第五行执行完毕之后，如果之后没有断点，则继续执行
    wathch i！=10
    info breakpoints 简写info break 会显示当前所有的断点，断点号断点位置。
    delete N 删除N号断点； delete 删除所有断点
    clear N 清除行N以上所有断点。
    whatis var 显示变量var的类型
