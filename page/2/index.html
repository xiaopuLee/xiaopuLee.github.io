首先简易说下caffe.cpp的使用方法

./build/tools/caffe train -solver examples/test/lenet_solver.prototxt -gpu 0
这是最简单的训练方法，其中-gpu 0 设定了device number, 如果不使用这个参数的话，程序会检查lenet_solver中是否指定mode 为 GPU，已经device num
./build/tools/caffe train -solver examples/test/lenet_solver.prototxt -gpu 0 -snapshot examples/test/lenet_solver_iter_75.solverstate
这是接着上次的训练，比如因为其他内在外在的原因，其中snapshot的参数为保存的以前的训练状态，例如上式中iter_75就是训练到75的时候
./build/tools/caffe train -solver examples/test/lenet_solver.prototxt -gpu 0 -weights examples/test/lenet_solver_iter_75.caffemodel
这是finetune的过程，其中.caffemodel是之前训练保存的参数。finetune真是好用，简单不到10次重复finetune就达到了我之前使用几万次迭代才能得到的结果

./build/tools/caffe device_query -gpu 0
查询制定的gpu的参数。

./build/tools/caffe test -model examples/test/lenet_train_test.prototxt -weights examples/test/lenet_solver_iter_75.caffemodel -gpu 0 -iterations 100
这是测试模型-model参数指的是神经网络模型的net参数 -weights 为训练出来的网络参数 -iterations为测试的迭代次数 默认为50

./build/tools/caffe time -model examples/test/lenet_train_test.prototxt -iterations 20
分析网络模型前向和后向传播的时间

static BrewFunction GetBrewFunction(const caffe::string& name) {
if (g_brew_map.count(name)) {
return g_brew_map[name];
} else {
LOG(ERROR) << “Available caffe actions:”;

今天来说一下google的开源库gflag

#用途
首先gflag是一个command line parser，是用来解析命令行输入参数的。
例如输入命令行 ./< your app > -year 2016 -month 4

#如何解析？
首先在全局初始化的时候可以这样定义一个变量
DEFINE_bool(flag, true, “help flag”);
上述语句定义了一个bool变量flag，默认值为true，变量帮助说明为 help flag的语句。在程序中，即可通过访问FLAGS_flag
这个变量来访问命令行穿进来的flag的值。是不是很方便？
除了bool值，gflag还支持其他5种数值类型，如int32 int64 uint64 string double。那么是否支持更复杂的类型呢？回答是no，google
给出的解释我觉得很赞–他们更愿意将对数据的处理放在程序中，而这时候gflag就只负责解析命令行的参数即可。
一段简易的代码。

#include<iostream>
#include<gflags/gflags.h>

DEFINE_bool(big_menu, true, "include 'advanced' options in the menu listing");
DEFINE_string(languages, "english, french, german", "comma-separated list of languages to offer int the 'lang' menu");

int main(int argc, char** argv)
{
   google::ParseCommandLineFlags(&argc, &argv, true);
   std::cout << FLAGS_languages << std::endl;
   if(FLAGS_big_menu)
   {
      std::cout << "the big menu is true" << std::endl;
   }
   else
   {
      std::cout << "the big menu is false" << std::endl;
   }
   return 0;
}

#如何通过command line传入参数呢？
所有的flag都可以有以下四中格式，如 –flag value， –flag=value -flag=value -flag value
bool除了以上四种格式，还可以通过–nobig_menu –big_menu来设定。

如果不小心同一个参数传入了多次呢？ 如./< your app > -year 2016 -month 4 -year 2013,此时以最好一个传入的为准。

#如何保证传入的参数的正确性呢？
首先我们需要在使用的时候就要确认好各种参数的类型以及他可能的取值范围。
其次，我们可以通过gflag的validator来做进一步的检测。
看代码。
static bool ValidatePort(const char* flagname, int32 value) {
if (value > 0 && value < 32768) // value is ok
return true;
printf(“Invalid value for –%s: %d\n”, flagname, (int)value);
return false;
}
DEFINE_int32(port, 0, “What port to listen on”);
static const bool port_dummy = RegisterFlagValidator(&FLAGS_port, &ValidatePort);
通常，上述validator需要在define 变量后立即register，需要在全局初始化的时候进行。分析上述代码即可知道他将port传入的值和validateport
函数绑定在一起，只要传入值即对这个值进行检验，例如在上述代码中，如果value 在（0， 32678）范围内，就会返回true，其他范围返回false，此时如果flag
是默认值的话，程序会go die，如果是其他值的话，程序会保留这个值。（待验证）

#其他

  1. 在ubuntu系统中，可以设置环境变量export FLAG_flag = 1 export FLAG_bar = 2
    然后–fromenv=flag,bar 即可从环境变量中读取值，但是环境中并未设置这样的值的话或者程序中并未设定这样的flag，程序会崩溃。一个折中的
    方法就是使用tryfromenv=flag，bar

    –languages=english,french
    然后通过–flagfile=f 即可访问。

使用gdb进行调试

#缘起
要在ubuntu下工作，又不太想把程序都放在window上写再来回转着调试，所以就想联系在linux系统里使用gdb进行调试

#相信使用

  1. 编译时加入debug信息, 使用参数-g即可。
    g++ -g main.cpp -0 main

  2. 启动测试
    可以使用gdb main，使用-tui 参数可以分屏显示代码，比参数l好用的要死
    也可以使用 gdb 打开调试窗口后，使用file app来打开app进行调试

    set args 来设定程序运行的参数
    也可以gdb之后，使用run <参数列表>
    show args显示参数

  3. 各种参数
    l list 列出当前行前后个5行代码 list n1 n2 列出n1 n2 之间的代码
    c continue 继续运行
    r run
    b break 设置断点 后面可以接行号 以及函数名字；也可以条件断点break 46 if testsize==100；break file：N 设置file的N行为断点
    p 打印后面变量的名字， print &var 打印变量的地址；如果a是一个数组，10个元素，如果要显示则： print *a@10
    n next 单步运行一次， next N，执行下N步

    s 跳入函数内部
    finish 执行完当前运行的函数，返回调用他的函数
    j jump 5 跳转到第五行执行完毕之后，如果之后没有断点，则继续执行
    wathch i！=10
    info breakpoints 简写info break 会显示当前所有的断点，断点号断点位置。
    delete N 删除N号断点； delete 删除所有断点
    clear N 清除行N以上所有断点。
    whatis var 显示变量var的类型

minist 数据经验
1 training data使用较大的batch 会相应增加训练时间，带来的利好是测试集的准确率会提升
原因很容易理解嘛，增加了batch的大小，本来就等于总训练数据迭代的次数增加了嘛，但是这种单一地增加maxIter并不一定
能够增加测试机的准确率。
https://www.zhihu.com/question/32673260 这里的讨论比较好
batch 256 max about 0.9920
batch 512 max 0.9931
batch 1024 max 0.9928 （相对于512时候，明显慢了很多很多，并且收敛的速度也变慢了好多）
